# Agile Method (애자일 방법론)

## 기존 방식: Waterfall / V-model

기존 방식은 순차적으로 개발 진행하여 (설계 → 구현 → 테스트) 테스트가 개발 후반에 수행된다. 이에 대해서 다음의 문제점들이 존재한다. 

- 테스트가 늦어 문제 발견 시 수정 비용 증가
- 개발 기간 예측 어려움
- 요구사항 변경 대응 어려움

---

## Agile Method란

Agile은 기능을 작은 단위로 나누어 반복적으로 개발하고, 지속적으로 검증하는 방법론이다.

### 특징
- 변화에 유연하게 대응
- 빠른 피드백
- 협업 중심 개발

| 구분 | Waterfall | Agile |
|------|----------|-------|
| 개발 방식 | 순차적 | 반복적 |
| 테스트 | 마지막 | 지속적 |
| 유연성 | 낮음 | 높음 |
| 리스크 | 큼 | 작음 |
---

## 핵심 개념

### Sprint

- 짧은 개발 주기 (보통 1~2주)
- 하나의 작은 기능을 완성하는 단위

각 Sprint는 다음 과정을 반복한다:

설계 → 구현 → 테스트 → 배포

---

### 작은 단위 개발

큰 프로젝트를 기능 단위로 나누어 개발한다.

예:
- 물류 자동화 시스템
  → 박스 인식
  → AMR 이동
  → 로봇팔 pick & place

---

### 지속적인 테스트

- 구현 후 바로 테스트 수행
- 문제를 빠르게 발견하고 수정

---

### Working Software

- 완벽한 코드보다 "동작하는 코드"가 중요
- 매 Sprint마다 실행 가능한 결과물을 만들어야 한다

---

## Scrum 방식

Scrum은 Agile을 구현하는 대표적인 방법이다.

### 구성 요소

- Product Owner: 요구사항 정의
- Scrum Master: 프로세스 관리
- Development Team: 개발 수행

---

### Scrum 흐름

Backlog 정의  
→ Sprint 계획  
→ 개발 (1~2주)  
→ 리뷰 및 회고  
→ 다음 Sprint

---

## 실무 도구

- Jira: 일정 및 Task 관리, Sprint 운영
- Confluence: 문서 및 회의 기록
- GitHub: 코드 관리 및 협업

---

## 버전 관리

v1.0: 첫 release  
v1.1: 버그 수정  
v1.x: 지속적인 개선

Agile에서는 자주 release하는 것이 중요하다.


---

## 프로젝트 적용 예시

Sprint 1: 박스 인식 (Vision)  
Sprint 2: AMR 이동  
Sprint 3: 로봇팔 제어  
Sprint 4: 전체 시스템 통합  

각 Sprint마다 실행 가능한 기능을 확보한다.
